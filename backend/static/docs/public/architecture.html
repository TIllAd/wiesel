Technische Architektur

Wie Wiesel technisch funktioniert

Eine kompakte Systemkarte: vom StudOn-Launch über die Chat-API bis zur Modellantwort. Keine Wand aus Techniktext — sondern Architektur, Datenfluss, Kostenkontrolle und Qualitätssicherung auf einen Blick.

Docker · Port 8001

Systemfluss

Studierende sehen nur den Chat. Im Hintergrund laufen Authentifizierung, Kontextaufbau, Modellaufruf, Logging und Qualitätsmarkierung getrennt voneinander.

Leserichtung: Launch/Auth → Backend → Kontext + Modell → Antwort + Logging.

Bausteine

Vier technische Entscheidungen, die das Projekt wartbar halten.

01

Wissensbasis

Die Fakten liegen als Markdown-Dateien in knowledge_base/. Inhalte können gepflegt werden, ohne die Chat-Oberfläche umzubauen.

02

System-Prompt

system-prompt.md trennt Charakter, Antwortstil und Sicherheitsregeln vom normalen Code.

03

Prompt Caching

Wissensbasis und Systemkontext werden cache-freundlich gesendet. Das senkt Latenz und Modellkosten bei wiederholten Anfragen.

04

Analytics

Tokens, Cache Reads/Writes, Kosten, Latenz und Fehler landen in llm_usage und können als Tagesexport ausgewertet werden.

Qualität und Grenzen

Der wichtigste technische Punkt: Wiesel soll helfen, aber keine falsche Sicherheit verkaufen.

Veraltete Fristen

Semesterspezifische Inhalte gehören in pflegbare Markdown-Dateien statt in fest verdrahteten Code.

Offizielle Quellen

Bei Prüfungen, Rücktritt, Krankheit, BAföG oder Rechtsfragen wird auf zuständige Stellen verwiesen.

Weniger Risiko

Der Bot gibt Orientierung, aber ersetzt keine verbindliche Auskunft.

Fehlerhafte Modellantwort

Auffällige Chats können markiert und später exportiert oder ausgewertet werden.

Logging ohne Neugier

Usage- und Fehlerdaten dienen Qualitätssicherung, nicht dem Stöbern in Rohchats.
